- Routes walked: 20 / 20 (plus 7 redirects)

- Blocker count: **3**
- High count: **5**
- Medium count: **5**
- Low count: **4**

**Most severe:** Workout builder emits segment distances in **meters** despite imperial preference, and paces are badly offset from the athlete's VDOT. That plus a complete lack of mobile navigation are the three blockers.

Unified athlete context is clearly wired — coach chat replied with the Tampa city I'd just saved to `/settings/profile` and referenced a specific recent activity by name. Settings save + preferences flip + unit swap across the app all work end-to-end. Dashboard hero, user menu, theme toggle, redirects, and every SSR page render without runtime errors.

## Findings

### [BLOCKER] Mobile — No navigation at any viewport < `md`
- **Steps:** Resize to 390 × 844, visit `/dashboard`, `/activities`, `/coach`.
- **Expected:** A hamburger, bottom tab bar, or collapsed nav drawer so the user can reach Coach / Activities / Dashboard / Plan / Insights / Teams / Settings.
- **Actual:** The sidebar is hidden (likely `hidden md:flex`) and nothing replaces it. There is no menu button, no bottom bar, no user-menu trigger. The only navigation on mobile is whatever in-content `<Link>` happens to be on the current page (e.g. "Manage →" on the dashboard, or the "Routes" tab on `/activities`). Users cannot cross-navigate.
- **Where:** `app/components/app-shell.tsx` (sidebar) + wherever `AppShell` is composed. No `<MobileNav>` is rendered.

### [BLOCKER] `/plan/build` — Segment distances render in meters despite imperial preference
- **Steps:** Settings preference = Imperial. Go to `/plan/build`, submit "Build me a tempo workout".
- **Expected:** `2 mi 18:00 @ 11:13/mi`, `4 mi @ 8:41/mi`, `1 mi 9:00 @ 11:13/mi`.
- **Actual:** `3219m 18:00 @ 11:13/mi`, `6437m @ 8:41/mi`, `1609m 9:00 @ 11:13/mi`. The LLM-produced JSON is in meters and the render component doesn't convert.
- **Where:** `app/app/(app)/plan/build/**` — workout-step card renders `segment.distance_m` verbatim. Need a units-aware formatter (same one the activity table uses).

### [BLOCKER] `/plan/race-sim` — Fixed `°C` and `m` labels regardless of preference
- **Steps:** Preferences = Imperial. Visit `/plan/race-sim`.
- **Expected:** `Temp (°F)` default `64`, `Elev gain (ft)` default `328`.
- **Actual:** `Temp (°C)` default `18`, `Elev gain (m)` default `100`. The form is hard-coded metric. The result card also likely renders metric.
- **Where:** `app/app/(app)/plan/race-sim/**` — the input labels and defaults are string literals; they need to read the athlete's units preference server-side.

### [HIGH] Workout builder — Paces are far off the athlete's VDOT
- **Steps:** VDOT is 51.2 (visible on `/dashboard` and `/insights/community`). Request a tempo workout on `/plan/build`.
- **Expected (Daniels VDOT 51):** Threshold ≈ 7:00/mi, Easy ≈ 8:53–9:26/mi.
- **Actual:** "Threshold pace (~8:41/mi based on VDOT 51.2)" — that's marathon/steady, not threshold. Warm-up + cool-down at 11:13/mi — that's slower than easy by ~2 min/mi (closer to walking pace).
- **Where:** `app/lib/ai/pace-derivation.ts` (or wherever VDOT → pace lookup lives), and the tool-call prompt for the builder. Either the VDOT lookup table is wrong, or the LLM is freelancing paces instead of being handed derived ones.

### [HIGH] Weekly-volume chart tooltip reads in km in imperial mode
- **Steps:** Imperial pref. Hover any bar on the `/dashboard` "Weekly volume" chart.
- **Expected:** `Distance : 31.7 mi` (or similar).
- **Actual:** `Distance : 51 km`. The rest of the tile's axis and values are in miles, so the tooltip is the only leak.
- **Where:** `app/components/dashboard/weekly-volume-chart.tsx` (Recharts `<Tooltip>` formatter) — the chart converts for the axis but not for the tooltip payload.

### [HIGH] `/settings/profile` — Weight input is labelled `(kg)` for imperial user
- **Steps:** Imperial pref. Open `/settings/profile`.
- **Expected:** `Weight (lbs)`; stored conversion happens server-side if needed.
- **Actual:** `Weight (kg)`. A value typed here will silently be interpreted as kg by anything downstream.
- **Where:** `app/app/(app)/settings/profile/page.tsx` — label is hard-coded.

### [HIGH] `/activities` — Mobile table overflows horizontally with no scroll affordance
- **Steps:** 390px viewport, visit `/activities`.
- **Expected:** Either a mobile-card layout, or a scroll container with overflow visible.
- **Actual:** The `DISTANCE`, `TIME`, `PACE`, `HR`, `LOAD` columns are clipped off the right edge. No `overflow-x-auto` scroll indicator, no touch scroll. The user just sees a truncated table.
- **Where:** `app/app/(app)/activities/page.tsx` + the activity list table component.

### [HIGH] `/plan/build` prompt chips/placeholder are half-metric in imperial mode
- **Steps:** Imperial pref. Visit `/plan/build`.
- **Expected:** All example prompts in the athlete's units.
- **Actual:** Placeholder "e.g. 4×1km at 5k pace with 2min jog between" and chip "6×1km at threshold with 90s jog between reps" are metric; "10 mile long run…" and "45 min recovery run…" are imperial. Inconsistent.
- **Where:** `app/app/(app)/plan/build/**` — hard-coded suggestion strings.

### [MEDIUM] Coach chat — Tempo paces echo the builder's incorrect VDOT table
- **Steps:** On `/coach`, ask "Build me a tempo workout".
- **Expected (VDOT 51.2):** Threshold reps around 7:00/mi.
- **Actual:** "4 × 1 mi @ 8:40–8:50/mi w/ 60s jog" — same slow-by-~100s/mi pattern as the builder. Because the coach has unified context (it did correctly reference Tampa + Friday's MP work), the pace lookup regression propagates everywhere VDOT-derived paces are used.
- **Where:** Shared pace derivation pathway — see BLOCKER #4.

### [MEDIUM] Dashboard — "6 runs" in the hero copy conflicts with the activity feed
- **Steps:** Visit `/dashboard`.
- **Expected:** Copy matches either (a) count of `Run` activities this week or (b) count of all activities.
- **Actual:** Hero says "28.9 mi across **6 runs**" while the Recent activities list this week shows 3 `Run` + 3 `WeightTraining`. Either the noun is wrong ("6 activities") or the count is counting non-runs.
- **Where:** `app/app/(app)/dashboard/page.tsx` hero verdict string.

### [MEDIUM] `/insights` time-machine — 2024/2023/2022 weeks show walking paces like 41:41/mi
- **Steps:** Visit `/insights`, look at "Time machine — week 16 through the years".
- **Expected:** Runs-only filter (or at minimum, flag non-running aggregates).
- **Actual:** Week 16 2024 row reads `4.2 mi · 41:41/mi`, which is a walk aggregated as if it were a run. Comparisons across years are useless if the sport mix changes.
- **Where:** `app/lib/analytics/time-machine.ts` (or equivalent) — should filter to `type = 'Run'`.

### [MEDIUM] Redirects preserve the legacy route in the URL bar briefly — fine — but no middleware 308 for internal RSC fetches
- **Steps:** Watch DevTools Network while visiting `/goal`.
- **Expected:** Single 308 → `/plan/goal`.
- **Actual:** Redirect works and qa-api validates 7/7, but a follow-up RSC prefetch fires for the OLD path, then gets redirected again. Cosmetic, but it's a wasted round-trip on every legacy link click. Not user-visible; medium because it will bite if deploys cache redirect responses.
- **Where:** `middleware.ts` — `matcher` should likely include RSC asset paths.

### [MEDIUM] A11y — Form inputs lack `name`, `id`, or `aria-label`
- **Steps:** Inspect `/settings/profile` inputs via accessibility snapshot.
- **Expected:** Each input has an associated `<label for>` or `aria-labelledby`.
- **Actual:** All 5 inputs have empty `name=""` and no `aria-label`. The text "First name", "Last name" etc. are rendered as sibling `<span>`s with no `htmlFor` wiring. Screen readers will read them as "edit text".
- **Where:** `app/app/(app)/settings/profile/page.tsx` — add `id` + `htmlFor`, or wrap in `<label>`.

### [LOW] qa-api.ts — Stale string assertion for `/dashboard`
- **Steps:** `node scripts/qa-api.ts <url>`.
- **Expected:** 29/29.
- **Actual:** 28/29. The one failure looks for literal `"Your training"`, which the hero now renders as "Today's read / Productively tired." The assertion is outdated; the page is fine.
- **Where:** `app/scripts/qa-api.ts:49` — update the expected string.

### [LOW] Recent activities widget — Everything today is labelled "Yesterday"
- **Steps:** Visit `/dashboard` on 2026-04-19.
- **Expected:** Activities from today → "Today"; from yesterday → "Yesterday".
- **Actual:** Four activities all say "Yesterday". Some were plausibly today-run-and-uploaded; without timestamps it's hard to be sure, but the widget also does the same on `/activities`. Either the relative-time helper rounds down aggressively, or the Strava timestamps are all ~20h old because the daily sync ran yesterday.
- **Where:** `app/lib/dates/relative.ts` or similar.

### [LOW] `/insights/community` — Percentile renders as a bare em-dash
- **Steps:** Visit `/insights/community`.
- **Expected:** Either a number, or a clearer empty state ("Waiting for more athletes to compare against").
- **Actual:** `WEEKLY MI PERCENTILE —` is confusing next to `25.8 mi/week (4-week avg)`. The footer explains "0 other athletes" but the top stat cards don't.
- **Where:** `app/app/(app)/insights/community/page.tsx`.

### [LOW] Dashboard chart console warnings on first paint
- **Steps:** Hard-reload `/dashboard`, open console.
- **Expected:** No warnings on load.
- **Actual:** Two Recharts warnings: `width(-1) and height(-1) of chart should be greater than 0…`. Charts render correctly once layout settles. Harmless but noisy.
- **Where:** Chart containers need `minHeight`/explicit height on first render.
